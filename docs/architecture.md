\# Kubernetes HPA Architecture on AWS EKS



\# Overview



This project demonstrates Kubernetes Horizontal Pod Autoscaler (HPA) implementation on Amazon Elastic Kubernetes Service (EKS).



The architecture includes:

\- Amazon EKS cluster

\- Worker nodes

\- Kubernetes Deployment

\- ClusterIP Service

\- Metrics Server

\- Metrics API

\- HPA Controller

\- Dynamic pod autoscaling



The implementation validates automatic scale-out and scale-in behavior based on CPU utilization metrics.

\# Architecture Components



\## 1. Amazon EKS Cluster



Amazon Elastic Kubernetes Service (EKS) provides the managed Kubernetes control plane.



The control plane components include:

\- kube-apiserver

\- etcd

\- scheduler

\- controller-manager



AWS manages the Kubernetes control plane automatically.



\---



\# 2. Worker Nodes



Worker nodes host the application pods.



Each worker node contains:

\- kubelet

\- container runtime

\- Kubernetes networking components

\- application pods



The worker nodes communicate with the Kubernetes control plane and execute scheduling instructions.



\---



\# 3. Kubernetes Deployment



The Deployment resource manages:

\- pod creation

\- replica management

\- rolling updates

\- self-healing



Deployment used in this project:



```text

hpa-demo-app

```



The deployment initially started with:

```text

1 replica

```



\---



\# 4. ClusterIP Service



A ClusterIP service exposes the application internally inside the Kubernetes cluster.



Purpose:

\- internal traffic routing

\- service discovery

\- load balancing between pods



Traffic generated by the load generator was routed through this service.



\---



\# 5. Pods



Pods host the NGINX application containers.



The pods were configured with:

\- CPU requests

\- CPU limits

\- memory requests

\- memory limits



Example:



```yaml

resources:

&#x20; requests:

&#x20;   cpu: "100m"

&#x20;   memory: "128Mi"



&#x20; limits:

&#x20;   cpu: "500m"

&#x20;   memory: "256Mi"

```



CPU requests are critical because HPA calculates utilization percentages using requested CPU values.



\---



\# 6. Metrics Server



Metrics Server collects resource metrics from kubelets running on worker nodes.



Collected metrics include:

\- CPU usage

\- memory usage



Metrics Server exposes metrics to:

```text

Metrics API

```



Without Metrics Server:

\- kubectl top fails

\- HPA cannot function



\---



\# 7. Metrics API



Metrics API exposes resource utilization data to Kubernetes components.



HPA continuously queries the Metrics API to evaluate:

\- current CPU usage

\- target utilization thresholds



Verification commands used:



```bash

kubectl top nodes

kubectl top pods

```



\---



\# 8. Horizontal Pod Autoscaler (HPA)



The Horizontal Pod Autoscaler automatically adjusts pod replica count based on CPU utilization.



HPA configuration:



| Parameter | Value |

|---|---|

| Minimum Replicas | 1 |

| Maximum Replicas | 10 |

| Target CPU Utilization | 50% |



The HPA monitored:

```text

Deployment/hpa-demo-app

```

\# HPA Decision Flow



The autoscaling lifecycle followed this sequence:



```text

Load Generator

&#x20;   ↓

ClusterIP Service

&#x20;   ↓

Deployment

&#x20;   ↓

Pods

&#x20;   ↓

Metrics Server

&#x20;   ↓

Metrics API

&#x20;   ↓

HPA Controller

&#x20;   ↓

Scale Decision

&#x20;   ↓

Additional Pods

```



\---



\# Autoscaling Formula



Horizontal Pod Autoscaler calculates desired replicas using:



DesiredReplicas = CurrentReplicas × (CurrentMetricValue / DesiredMetricValue)



Observed example during testing:



```text

DesiredReplicas = 1 × (148 / 50) ≈ 2.96

```



Kubernetes progressively scaled the deployment as CPU utilization remained above the configured threshold.



\---



\# Autoscaling Behavior Observed



During load testing:

\- CPU utilization increased significantly

\- HPA detected threshold violations

\- Additional replicas were created automatically



Observed scaling progression:



```text

1 → 2 → 3 → 4

```



Peak CPU utilization observed:



```text

148%

```



After scaling:

\- traffic distribution improved

\- CPU utilization reduced

\- workload stabilized automatically



Observed CPU reduction:



```text

58% → 44% → 31% → 15% → 0%

```



\---



\# Scale-In Behavior



After load generation stopped:

\- CPU utilization dropped

\- HPA stabilization logic delayed immediate scale-in

\- replicas gradually reduced



Observed scale-in lifecycle:



```text

4 → 3 → 2 → 1

```



This demonstrated:

\- stabilization window behavior

\- prevention of scaling flapping

\- gradual workload normalization



\---



\# Kubernetes Stabilization Logic



Kubernetes does not immediately remove replicas after CPU utilization drops.



Reason:

```text

Prevent scaling flapping

```



Flapping refers to:

\- rapid scale-out

\- immediate scale-in

\- unstable scaling behavior



HPA stabilization windows improve cluster stability.

\# Key Architecture Learnings



\- Metrics Server is mandatory for HPA operations

\- HPA depends on Metrics API availability

\- CPU requests directly influence scaling behavior

\- Kubernetes scaling is gradual, not instantaneous

\- Traffic distribution reduces CPU utilization after scaling

\- HPA stabilization prevents scaling instability

\- Kubernetes automatically reconciles desired replica states
